# 🔒 Protected System Boundary

## Purpose

This repository exposes only the **customer-facing assurance surface** of the ELYRIA SYSTEMS dependency-boundary method.

It does **not** expose the protected kernel, proprietary substrate, internal execution logic, internal scoring rationale beyond customer-usable thresholds, private research corpus, or non-public system architecture.

---

## What This Repository May Expose

The repository may expose:

- customer-facing intake forms
- board-facing review language
- dependency-boundary categories
- public pilot/demo materials
- lightweight sandbox behavior
- example receipts
- high-level scoring structure
- buyer-facing decision outcomes
- non-sensitive templates

---

## What Must Remain Protected

The repository must not expose:

- protected kernel internals
- proprietary consequence-boundary substrate
- private Elyria Systems architecture
- non-public mathematical derivations
- internal enforcement logic
- private source corpus
- unpublished research notes
- customer confidential data
- real customer dependency evidence
- production secrets, API keys, tokens, credentials, or environment values
- privileged implementation details that enable cloning of the full system

---

## Sandbox Rule

The customer sandbox is a **demonstration surface**, not the protected system.

It should show:

- how a dependency is submitted
- how a board-facing outcome is returned
- how a receipt is generated
- how revalidation triggers are surfaced

It should not reveal the full protected methodology.

---

## Public Boundary Statement

**The customer receives assurance.**  
**The board receives a decision object.**  
**The protected substrate remains sealed.**

---

## Operational Rule

Expose enough for customers to understand, test, and buy the review.

Do not expose enough for competitors to reconstruct the protected Elyria Systems substrate.
